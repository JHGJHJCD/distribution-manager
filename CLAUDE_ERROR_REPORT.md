# דוח בדיקה מקיפה עבור Claude - מנהל חלוקה

תאריך בדיקה: 2026-05-31  
תיקייה: `מנהל_חלוקה`  
הערת משתמש חשובה: קובץ Excel הוא דוגמה בלבד, ולכן חוסרים בנתוני הדוגמה אינם שגיאת תוכנה בפני עצמם.

## תקציר מנהלים

האפליקציה במצב טוב יותר מהבדיקות הקודמות: היא נטענת, המודולים עוברים import, המסכים המרכזיים נוצרים, ייבוא Excel עובד, ותיקונים קודמים כמו שמירת הערות, חסימת מחיקת מקבל עם היסטוריה, ותאריכים בייבוא כבר קיימים.

הממצאים החשובים כעת הם בעיקר תפעוליים:

1. `test_all.py` מסוכן כי הוא מוחק את `data.db` האמיתי אם מריצים אותו מתיקיית האפליקציה.
2. בגרסת EXE מסוג PyInstaller onefile יש סיכון גבוה שה-DB/ייצוא/גיבוי יפנו לתיקיית חילוץ זמנית במקום ליד קובץ ה-EXE.
3. במסד הנוכחי יש רשומת בדיקה `__test__` וחלוקת בדיקה. לא למחוק בלי אישור, אבל זה לא אמור להיות בנתוני אמת.
4. ייבוא Excel עובד, אבל צריך להפוך אותו לידידותי יותר: דוח כפילויות, שדות חסרים, ומה בדיוק עודכן.
5. נוחות השימוש טובה כבסיס, אך חסרים אשף התחלה, בדיקת איכות נתונים, bulk edit, סינון מתקדם, והבהרה של זרימות עבודה.

## ממצאים קריטיים

### 1. `test_all.py` מוחק את מסד הנתונים האמיתי

חומרה: קריטית  
קובץ: `test_all.py:24-25`

בעיה:

```python
if os.path.exists("data.db"):
    os.remove("data.db")
```

אם מריצים את הבדיקה מתיקיית האפליקציה, היא מוחקת את `data.db` האמיתי. בנוסף הבדיקה מוסיפה מקבל `__test__` וחלוקה, ולכן היא יכולה להשאיר נתוני בדיקה במסד.

השפעה:

- אובדן נתונים אמיתיים.
- זיהום נתוני ייצור ברשומות בדיקה.
- חוסר אמון בבדיקות.

תיקון מומלץ:

- להשתמש תמיד ב-DB זמני:

```python
import tempfile
fd, path = tempfile.mkstemp(suffix=".db")
os.close(fd)
db.DB_PATH = path
```

- לא למחוק `data.db` אף פעם מתוך טסט.
- להוסיף guard:

```python
assert Path(db.DB_PATH).name != "data.db"
```

- להוסיף cleanup בסוף, אחרי סגירת כל החיבורים.

### 2. נתיבי DB/ייצוא/גיבוי בעייתיים ב-PyInstaller onefile

חומרה: קריטית להפצה כ-EXE  
קבצים:

- `database.py:6`
- `utils/excel_utils.py:220`
- `utils/backup.py:14`
- `main.py:5-8`
- `build_exe.bat:7`

בעיה:

`main.py` משנה working directory ליד ה-EXE, אבל `database.py` קובע:

```python
DB_PATH = os.path.join(os.path.dirname(__file__), "data.db")
```

ב-PyInstaller onefile, `__file__` של המודול יכול להצביע לתיקיית חילוץ זמנית. גם `excel_utils.py` משתמש ב-`Path(__file__)`, ו-`backup.py` משתמש ב-`db.__file__`.

השפעה אפשרית:

- `data.db` יישמר בתיקייה זמנית וייעלם/יתאפס בהפעלה הבאה.
- ייצוא יישמר לתיקייה לא צפויה.
- גיבוי יעתיק את ה-DB הלא נכון.
- המשתמש יחשוב שהנתונים נמחקו.

תיקון מומלץ:

להגדיר פונקציה מרכזית אחת:

```python
def app_dir():
    if getattr(sys, "frozen", False):
        return Path(sys.executable).resolve().parent
    return Path(__file__).resolve().parent
```

ואז:

- `DB_PATH = app_dir() / "data.db"`
- export ל-`app_dir() / "exports"`
- backup יעתיק מ-`database.DB_PATH`, לא מ-`db.__file__`.

### 3. אין מערכת מיגרציות ל-DB קיים

חומרה: גבוהה  
קובץ: `database.py:init_db`

בעיה:
`CREATE TABLE IF NOT EXISTS` לא משנה טבלאות קיימות. אם למשתמש כבר יש `data.db` ישן בלי עמודה/constraint חדש, הקוד לא יעדכן אותו.

דוגמה:
אם `distributions` נוצר בעבר בלי `ON DELETE SET NULL`, `init_db()` לא יוסיף אותו.

תיקון מומלץ:

- להוסיף `PRAGMA user_version`.
- להריץ מיגרציות מסודרות לפי גרסה.
- לבדוק `PRAGMA table_info` ו-`PRAGMA foreign_key_list`.
- לבצע גיבוי לפני כל מיגרציה.

### 6. `get_weekly_list` מחשב תאריך זמני אך לא שומר אותו

חומרה: בינונית

בעיה:

השפעה:

- המשתמש רואה תאריך במסך, אבל בחיפוש/עריכה/DB השדה עדיין חסר.
- אחרי שינוי לוגיקה או תאריך, תוצאות עשויות להיראות לא עקביות.

תיקון מומלץ:

- להציג תגית "מחושב" ליד תאריך שלא נשמר.
- או להוסיף פעולה "מלא תאריכי חלוקה חסרים".
- או לעדכן DB כאשר התאריך מחושב, עם הודעת אישור.

### 7. חד-פעמי מקבל `next_distribution` אחרי חלוקה

חומרה: בינונית-נמוכה  
קובץ: `database.py:263-282`

בעיה:
`bulk_add_distributions` מחשב `next_distribution` גם לחד-פעמי. מאחר שחד-פעמיים מוחרגים מרשימת השבוע, זה לא מפיל את המערכת, אבל סמנטית זה מבלבל.

תיקון מומלץ:

- עבור `frequency == "חד-פעמי"` לקבוע `next_distribution=''`.
- או להציג בטאב חד-פעמי רק `last_distribution` ולא להשתמש ב-`next_distribution`.

### 8. מספרי טלפון עם 10 ספרות שמתחילים ב-5 לא מתוקנים

חומרה: בינונית

מצב:

ייתכן שזה תקין אם הנתון בדוגמה באמת כך, אבל בטלפון ישראלי רגיל זה נראה חשוד.

תיקון מומלץ:

- להוסיף דוח "מספרים חשודים" אחרי ייבוא.
- לא לתקן אוטומטית 10 ספרות בלי ודאות, אלא לסמן לבדיקה.

## המלצות תיקון לפי עדיפות

1. לתקן את `test_all.py` כך שיעבוד רק מול DB זמני.
2. לתקן נתיבי DB/export/backup ל-EXE עם `sys.frozen` ו-`sys.executable`.
3. להסיר/לנקות רשומות בדיקה מה-DB אחרי גיבוי ואישור משתמש.
4. להוסיף דוח איכות נתונים אחרי ייבוא.
5. להוסיף מיגרציות DB עם `user_version`.
6. להוסיף מסך התחלה או אשף "ייבוא והשלמת נתונים".
7. להוסיף bulk edit למקבלים.
8. להבהיר/לשמור את סימון הביצוע בטאב השבועי.
9. לשפר חיפוש ומעקב עם סינוני סטטוס/תאריך.
10. להעביר הגדרות לטאב ייעודי ולהציג מצב גיבוי.

## בדיקות מומלצות אחרי תיקון

- להריץ בדיקות מול DB זמני בלבד.
- לבנות EXE ולבדוק:
  - נתונים נשמרים אחרי סגירה ופתיחה מחדש.
  - `data.db` נמצא ליד ה-EXE.
  - ייצוא נשמר ב-`exports` ליד ה-EXE.
  - גיבוי מעתיק את ה-DB הנכון.
- לבדוק ייבוא קובץ דוגמה וקובץ אמת.
